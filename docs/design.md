# 设计文档 · Prompt Optimizer

## 1. 项目概述

### 1.1 项目背景
用户在使用 AI 时，经常因为提示词（Prompt）写得不够清晰、结构混乱、缺乏约束，导致 AI 输出质量不稳定。一个好的提示词能够显著提升 AI 的产出质量，但大多数用户缺乏系统的提示词工程知识。

### 1.2 项目目标
打造一个提示词优化 Skill，帮助用户快速将简单、模糊的原始需求，转化为结构化、专业化、目标明确的高质量提示词，从而提升 AI 输出的准确性和质量。

### 1.3 核心价值
- **降低门槛**：用户无需学习提示词工程，直接获得专业级 prompt
- **提升效率**：一键优化，节省用户反复调试 prompt 的时间
- **质量保障**：基于成熟框架，输出结构化、可复用的高质量 prompt
- **灵活可控**：支持多框架选择、对比展示、用户修改确认

## 3. 三大框架设计

### 3.1 CRISPE 框架（全面型）

**设计思路**：参考业界成熟的 CRISPE 提示词框架，针对中文使用场景进行优化调整。

**适用场景**：
- 复杂任务、多步骤任务
- 需要精确控制输出质量
- 涉及专业领域知识
- 需要可复现的稳定输出

**六维度说明**：

| 维度 | 全称 | 作用 | 常见内容 |
|------|------|------|---------|
| C | Capacity & Role | 设定AI身份 | 专业角色、经验背景、能力范围 |
| R | Background & Context | 提供背景 | 任务背景、前置信息、上下文 |
| I | Input Data & Instructions | 明确输入与指令 | 输入数据、具体任务、执行步骤 |
| S | Output Style & Tone | 规定风格 | 语气、文风、视角、格式 |
| P | Performance Goal & Constraints | 目标与约束 | 输出目标、限制条件、禁忌 |
| E | Example & Evaluation | 示例与评估 | 参考示例、评估标准、验收要求 |

### 3.2 ROSES 框架（创意/文案型）

**设计思路**：针对文案写作和创意生成类任务的特点，设计专门的五维框架，强调受众、场景、风格等创作要素。

**适用场景**：
- 文案写作（公众号、小红书、广告语等）
- 创意生成（故事、命名、活动策划等）
- 品牌营销类内容
- 需要考虑受众心理的内容

**五维度说明**：

| 维度 | 全称 | 作用 | 常见内容 |
|------|------|------|---------|
| R | Role | 创作身份 | 资深文案、创意总监、小说家等 |
| O | Objective | 创作目标 | 传播目标、转化目标、情感目标 |
| S | Scenario & Audience | 场景与受众 | 使用场景、目标受众、受众画像 |
| E | Elements & Constraints | 要素与约束 | 必含要素、风格要求、字数限制 |
| S | Style & Structure | 风格与结构 | 文风调性、篇章结构、输出格式 |

### 3.3 结构化多层优化（灵活通用型）

**设计思路**：对于通用任务，采用更灵活的七层结构，用 emoji 增强可读性，让用户一目了然。

**适用场景**：
- 通用任务
- 目标单一的简单任务
- 需要灵活调整维度的场景
- 用户不确定用哪个框架时的默认选择

**七维度说明**：

| 维度 | emoji | 作用 |
|------|-------|------|
| 核心目标 | 🎯 | 一句话明确任务目的 |
| 角色设定 | 👤 | AI 应扮演的身份 |
| 任务清单 | 📋 | 拆解为具体子任务 |
| 约束条件 | 🚫 | 不能做什么、必须遵守的规则 |
| 输出格式 | 📐 | 结构、长度、格式要求 |
| 参考示例 | 💡 | Good/Bad 示例或参考方向 |
| 质量标准 | 📊 | 评估输出好坏的标准 |

## 4. 自动识别机制

### 4.1 识别策略
采用**关键词匹配 + 场景分类**的双层识别策略：

```
输入原始 prompt
    │
    ▼
第一层：关键词匹配
    ├─ 匹配文案类关键词 → 文案场景
    ├─ 匹配创意类关键词 → 创意场景
    └─ 无匹配 → 通用场景
    │
    ▼
第二层：复杂度判断
    ├─ 多步骤/多要求 → CRISPE（全面型）
    ├─ 文案/创意场景 → ROSES（创意型）
    └─ 简单/通用任务 → 结构化多层优化
```

### 4.2 关键词库

**文案类关键词**：
文案、文章、推文、公众号、小红书、种草、广告语、slogan、标语、短视频、脚本、口播、邮件、通知、公告、邀请函、软文、推文、标题

**创意类关键词**：
创意、故事、小说、诗歌、歌词、命名、取名、起名、策划、方案、活动、灵感、脑洞、创作、剧本、剧情

**复杂任务特征**：
- 包含多个步骤（首先、然后、最后、第一步等）
- 涉及专业领域（代码、法律、医疗、金融等）
- 有明确的格式要求（表格、JSON、特定结构）
- 字数较多、要求较细

## 5. 交互设计

### 5.1 交互流程图

```
用户输入 /optimize [内容]
    │
    ▼
检测内容是否为空
    ├─ 空 → 询问用户输入内容 → 返回
    └─ 非空 → 继续
    │
    ▼
检测是否指定框架
    ├─ 是 → 使用指定框架
    └─ 否 → 自动识别框架 + 告知用户
    │
    ▼
生成优化后的 prompt
    │
    ▼
对比展示（原始 vs 优化后 + 优化亮点）
    │
    ▼
用户选择操作
    ├─ 确认使用 → 输出干净版本 + 询问是否执行
    ├─ 修改调整 → 接收修改意见 → 重新优化 → 回到对比展示
    ├─ 切换框架 → 用新框架优化 → 回到对比展示
    └─ 直接执行 → 作为新任务执行 → 结束
```

### 5.2 对比展示设计

**设计原则**：
1. 清晰对比：左右/上下对比，一目了然
2. 亮点说明：告诉用户好在哪里，教育用户
3. 框架标注：标明使用的框架，增加专业感
4. 操作便捷：提供清晰的下一步操作选项

**展示结构**：

## 7. 优化原则与质量标准

### 7.1 七大优化原则

1. **目标明确原则**
   - 优化后的 prompt 必须有清晰的目标
   - AI 一看就知道要做什么、做到什么程度

2. **角色具体原则**
   - 角色设定要专业、具体
   - 避免笼统的"你是一个助手"
   - 增加经验背景、专业领域等细节

3. **约束清晰原则**
   - 明确什么能做、什么不能做
   - 明确输出格式、字数、风格要求
   - 列出禁忌和注意事项

4. **结构分层原则**
   - 使用编号、标题、符号等分层
   - 让 prompt 结构清晰易读
   - 重要内容加粗或高亮

5. **示例引导原则**
   - 适当提供示例或参考方向
   - 用 Good/Bad 示例减少歧义
   - 示例要典型、有代表性

6. **保留原意原则**
   - 优化是增强表达，不是改写需求
   - 必须完整保留用户的原始意图
   - 不添加用户没有提到的要求

7. **语言一致原则**
   - 优化后的 prompt 语言与输入保持一致
   - 中文输入 → 中文优化
   - 英文输入 → 英文优化

### 7.2 质量评估标准

| 评估维度 | 优秀标准 | 合格标准 | 不合格 |
|---------|---------|---------|--------|
| 目标清晰度 | 一句话就能看懂要做什么 | 基本能理解任务 | 看不懂要做什么 |
| 结构完整性 | 涵盖所有必要维度 | 主要维度齐全 | 缺少关键信息 |
| 可执行性 | AI 能直接产出高质量结果 | AI 能产出可用结果 | AI 输出质量差 |
| 无歧义性 | 没有任何模糊表述 | 基本没有歧义 | 有多处模糊表述 |
| 专业度 | 角色设定专业、细节丰富 | 角色设定合理 | 角色笼统模糊 |

## 8. 技术实现说明

### 8.1 Skill 机制
本 Skill 基于 TRAE 的 Skill 系统实现，通过 `SKILL.md` 文件定义触发条件、行为规范和操作指南。

### 8.2 实现方式
- **纯提示词驱动**：所有优化逻辑通过 SKILL.md 中的指令实现，无需额外代码
- **LLM 内执行**：利用大模型本身的理解和生成能力完成优化
- **模板库参考**：优化时可参考 templates/ 目录下的专业模板

### 8.3 触发方式
1. **前缀触发**：`/optimize` 前缀（主要方式）
2. **语义触发**：用户明确要求优化 prompt 时自动触发
